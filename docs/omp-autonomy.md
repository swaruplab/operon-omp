# OMP Autonomy Features in Operon

Operon swaps its agent engine from OpenCode to **OMP** (oh-my-pi, package
`@oh-my-pi/pi-coding-agent`, MIT — <https://github.com/can1357/oh-my-pi>) to get
a more autonomous, more robust agent "brain." OMP ships as a self-contained
binary (`curl -fsSL https://omp.sh/install | sh` → ~150 MB to `~/.local/bin`, no
root, no Bun/Node at runtime; macOS arm64/x64, Linux x64/arm64, Windows x64).

The engine is selected by the `agent_engine` setting (Rust:
`AppSettings.agent_engine` in `src-tauri/src/commands/settings.rs`), which
defaults to `"omp"`. Set it to `"opencode"` for one-flag rollback to the legacy
adapter.

Operon drives OMP as a one-shot CLI: `omp --mode json -p …` streaming JSONL
events (see [`omp-event-schema.md`](./omp-event-schema.md)). The harness adapter
is `src-tauri/src/harness/omp.rs`. On first run it generates OMP's global config
(`~/.omp/agent/{models,config}.yml`) and a guardrail hook, via
`ensure_omp_config()` (idempotent and non-destructive — it never overwrites a
user's hand-tuned files).

This document describes the **four autonomy capabilities** Operon wires in and
**where each lives**.

> **Status — honest scope.** All four are configured and the core swap compiles
> (`cargo check` green), grounded against the real `omp` v16.0.5 binary locally.
> The **HPC remote path has not yet been validated on a real cluster** — the
> gating test is glibc compatibility of the Linux binary on the target nodes
> plus a one-shot `omp --mode json` tailed across the login/compute split.

## 1. Model-role routing + fallback chains

**What it gives us.** The agent routes sub-tasks to the right model by *role*
instead of pinning one model, and automatically falls back to another
provider/model when a call fails — so a flaky local server or a context-length
error degrades gracefully instead of killing the run.

**Where it's wired.** `ensure_omp_config(model)` in `src-tauri/src/harness/omp.rs`
generates `~/.omp/agent/config.yml` with two relevant blocks:

```yaml
# ~/.omp/agent/config.yml  (generated by ensure_omp_config)
modelRoles:
  default: ollama/qwen3:32b   # the model Operon was launched with
  plan:    ollama/qwen3:32b
  smol:    ollama/qwen3:32b
  slow:    ollama/qwen3:32b
  commit:  ollama/qwen3:32b
retry:
  enabled: true
  fallbackChains:
    default: [ollama/qwen3:32b]
```

- `modelRoles` maps OMP's roles (`default`, `plan`, `smol`, `slow`, `commit`) to
  concrete `provider/model` ids. By default Operon points every role at the one
  model the user selected, which is a safe, single-model starting point.
- `retry.fallbackChains` lists, per role, the ordered providers/models OMP tries
  on failure.

**How to take advantage of it.** Edit `~/.omp/agent/config.yml` by hand — Operon
will not overwrite it. Point `smol` at a fast/cheap local model for trivial
sub-steps, `slow`/`plan` at a stronger reasoning model, and add real alternates
to each `fallbackChain` (e.g. a second Ollama host, an LM Studio endpoint, or a
cloud model). Provider definitions and base URLs live in the sibling
`~/.omp/agent/models.yml`, also generated on first run (OpenAI-compatible
`baseUrl` per provider; default `http://localhost:11434/v1` for Ollama).

Fallbacks are observable in the event stream as `auto_retry_*` /
`retry_fallback_*` events (surfaced as `raw` by the adapter).

## 2. Programmable guardrails (hooks + TTSR)

Two complementary layers, both important because Operon runs OMP **headless**
(stdout redirected to a file, tailed over SSH). There is no TTY and no
interactive approval prompt — so destructive actions must be *hard-blocked*, not
"confirmed."

### 2a. Pre-tool-call hook (hard block)

`ensure_omp_config` writes a bundled TypeScript hook to
**`~/.omp/hooks/pre/operon-guardrails.ts`** (source is the `GUARDRAILS_HOOK_TS`
constant in `src-tauri/src/harness/omp.rs`). It runs on OMP's `tool_call` hook
and, for `bash`/`shell` tools, hard-blocks:

1. `rm` targeting a **shared/data filesystem** path
   (`/dfs*`, `/data`, `/gpfs`, `/scratch`, `/nfs`, `/projects`), and
2. recursive delete of a **top-level path** (`rm -rf /`, `~`, `$HOME`).

A blocked call returns `{ block: true, reason: "Operon guardrail: …" }` so the
agent sees the refusal and can choose a safer action. The hook is regenerated on
first run if absent and is never overwritten, so labs can extend it with their
own rules (e.g. block writes outside a project root, or quarantine a controlled
dataset mount).

### 2b. TTSR (time-traveling streamed rules)

TTSR rules are regex-triggered reminders injected into the agent's context the
moment matching content streams by — a lightweight, declarative complement to
the hard-block hook. They live as markdown rule files under **`.omp/rules/`**.
Operon ships a lab template at
[`.omp/rules/operon.ttsr.md`](../.omp/rules/operon.ttsr.md) that:

- on controlled-access data terms (`dbGaP|controlled-access|PHI|patient.identifi`)
  injects a reminder to keep such data on **local** models, and
- on `sbatch` injects the **submit-and-end-turn** rule (don't poll `squeue`).

Adapt that template per project. When a rule fires, OMP emits a
`ttsr_triggered` event (surfaced as `raw`).

### Headless reinforcement (system prompt)

The same intent is reinforced from the Rust side via `HEADLESS_RULES`
(appended to the system prompt every turn in `build_command`): no TTY/UI waits,
prefer non-interactive flags, and the "submit `sbatch` then end the turn — the
watchdog tracks completion" rule. Operon's permission tiers (`safe_mode`,
`supervised`, `full_auto`) are layered on top here as additional system-prompt
guidance, while `--auto-approve` keeps the headless run from blocking on
approval prompts.

## 3. Subagent swarm

**What it gives us.** OMP can spawn isolated subagents to parallelize a task
(e.g. one subagent per file, per analysis, or per hypothesis) via its `task` /
`agents` tooling. This is **enabled by default** — no extra Operon config is
required, and the spawning agent coordinates the swarm itself.

**Isolation backends.** OMP isolates each subagent's working tree so they don't
clobber each other. The available backends are:

- **`worktree`** — git worktrees (portable; the safe default on HPC/Linux).
- **`fuse`** — a FUSE overlay (Linux, where FUSE is available).
- **`ProjFS`** — Windows Projected File System (Windows hosts).

On HPC the `worktree` backend is the dependable choice since FUSE is often
unavailable on compute nodes. Subagent activity appears in the stream as the
normal `agent_*` / `tool_execution_*` events; Operon renders the resulting tool
calls and assistant text the same way it renders the primary agent.

## 4. Hashline edits + persistent Python kernel

### 4a. Hashline edits (default)

OMP's default file-edit strategy is **hashline edits**: edits are anchored to
content hashes of the surrounding lines rather than to raw line numbers or a
fragile search/replace string. This makes edits far more robust to a file having
shifted since the model last read it — a common failure mode for line-based
patching in long agent runs. Nothing to configure; it is the default edit tool.

### 4b. Persistent Python kernel

OMP can run a **persistent Python kernel** so the agent keeps variables, imported
libraries, and loaded dataframes alive *across* tool calls within a session —
instead of paying cold-start + re-load on every snippet. This is ideal for
bioinformatics workflows that load a large matrix once and then iterate on it.

Enable it once per host with:

```bash
omp setup python
```

After setup, the agent's Python tool reuses the live kernel. On HPC, run
`omp setup python` inside the conda/venv the analysis needs (so the kernel sees
the right `scanpy`/`pandas`/etc.), on a node that shares the project filesystem.

## Quick reference: where each capability is configured

| Capability | Configured in | Owned by |
|---|---|---|
| Model-role routing | `~/.omp/agent/config.yml` → `modelRoles` | `ensure_omp_config` |
| Fallback chains | `~/.omp/agent/config.yml` → `retry.fallbackChains` | `ensure_omp_config` |
| Provider/base URLs | `~/.omp/agent/models.yml` | `ensure_omp_config` |
| Guardrail hard-block | `~/.omp/hooks/pre/operon-guardrails.ts` | `GUARDRAILS_HOOK_TS` |
| TTSR rules | `.omp/rules/*.ttsr.md` | lab template `operon.ttsr.md` |
| Headless + HPC rules | system prompt (`HEADLESS_RULES`) | `build_command` |
| Subagent swarm | OMP built-in (default on) | OMP (`task`/`agents` tools) |
| Hashline edits | OMP built-in (default edit tool) | OMP |
| Persistent Python kernel | `omp setup python` (per host) | OMP |
| Engine selection / rollback | `agent_engine` setting (`"omp"`/`"opencode"`) | `AppSettings` |
